01 // Foundations

What is an Agentic System?

An AI Agent is not a chatbot with memory. It is an autonomous reasoning engine that perceives its environment, decides which actions to take, executes those actions via tools, and iterates toward a goal — without a human directing every step.

🤖 Traditional Chatbot

Input → LLM → Output. One turn. No memory of past turns beyond the window. No action capability. No goal pursuit.

⚡ AI Agent

Perceive → Think → Plan → Act → Observe → Repeat. Multi-turn, goal-driven, tool-calling, stateful across sessions, capable of real-world side effects.

🕸️ Multi-Agent System

Orchestrator agent delegates to specialist agents (search agent, DB agent, checkout agent). Each agent owns a domain. Output is collaborative.

02 // Comparison

Traditional Software Design vs. Agentic System Design

Both solve business problems. But the architecture, control flow, and failure modes are fundamentally different. Here's how a senior software engineer should reframe their mental model.

Dimension Traditional Software System Agentic AI System
Control Flow Deterministic. Code dictates every branch. You own the flow graph completely. Semi-deterministic. LLM decides next action at runtime. You design guardrails, not exact paths.
State Management Stored in DB, session, cache. Explicit reads/writes by you. State = context window + external memory. Agent manages its own working memory each call.
Input Handling Typed, validated, structured schema (REST/gRPC). Natural language (unstructured). Intent must be inferred and extracted before acting.
Business Logic Hardcoded in functions, classes, services. Expressed as instructions in system prompt + few-shot examples + tool definitions.
Integration REST APIs, DB drivers, message queues. Explicit contracts. Tools (functions the LLM can call). MCP servers. Agent decides when/how to invoke them.
Error Handling try/catch, circuit breakers, retry logic in code. Error fed back into context. Agent reasons about failure and self-corrects. You set retry limits.
Testing Unit tests, integration tests, E2E. Pass/fail is deterministic. Eval frameworks (LLM-as-judge, human eval, golden datasets). Probabilistic correctness.
Deployment Unit Microservice, container, serverless function. Agent = LLM + system prompt + tools + memory. Deployed as a reasoning loop service.
Scalability Horizontal scaling of stateless services. Parallelise agent instances. Context window is the bottleneck, not compute.
Observability Logs, metrics, traces (OpenTelemetry). Token traces, tool call logs, context snapshots, latency per reasoning step.
Security Auth, RBAC, input sanitisation, SQL injection prevention. Prompt injection, tool misuse guardrails, PII redaction, consent gates before side effects.
Failure Mode Exception, null pointer, timeout, race condition. Hallucination, context rot, infinite loops, tool misuse, goal misalignment.

03 // Core Components

Anatomy of a Production AI Agent

Every production agent is composed of six core components. Each maps to a familiar software concept — but with a twist.

🧠
LLM (The Reasoner)
The CPU. Reads context, reasons, decides next action or generates final response. Does NOT store state between calls.
claude-sonnet / gpt-4o / gemini-pro
Model selection = capability vs. cost tradeoff
📋
System Prompt (The Config)
Personality, role, rules, tone, constraints. Your software equivalent: config + business logic combined. This IS your agent's behaviour.
Role definition + tone + tool instructions
+ few-shot examples + guardrails
🔧
Tools (The Connectors)
Functions the LLM can invoke. Search, DB read/write, API calls, cart operations. The agent decides WHEN and HOW to call them.
search_products() | add_to_cart()
update_inventory() | get_recommendations()
💾
Memory System (The State)
Short-term: conversation history in context. Long-term: vector DB for semantic recall. Episodic: structured records of past sessions.
Redis (short-term) | Pinecone/pgvector (long-term)
PostgreSQL (episodic/structured)
🔍
RAG Pipeline (The Knowledge)
Retrieval-Augmented Generation. Inject relevant product catalog, policy docs, FAQs into context at runtime. Dynamic knowledge injection.
Embed → Index → Retrieve → Inject
LlamaIndex / LangChain / custom
🛡️
Guardrails (The Safety Layer)
Input/output validation. PII detection. Consent gates before side effects (checkout). Hallucination detection. Loop limits.
NeMo Guardrails | custom validators
Prompt injection filters | rate limits
📊
Observability (The Monitoring)
Trace every reasoning step, every tool call, every token. Not just "did it work?" but "why did the agent take that path?"
LangSmith | Langfuse | OpenTelemetry
Token usage, latency, tool call logs
🔄
Orchestrator (The Controller)
Manages the reasoning loop. Routes sub-tasks to specialist agents. Controls context size. Handles retries and escalation.
LangGraph | CrewAI | custom loop
Async queues for long-running tasks

04 // Core Pattern

The Agent Reasoning Loop

This is the heartbeat of every agent. Unlike a REST handler that runs once, the agent loop iterates until it reaches a stopping condition. Understanding this loop is understanding agents.

👁️
PERCEIVE
Receive user input. Retrieve relevant memory. Assemble context window.
🗂️
CONTEXT WINDOW
System prompt + history + retrieved knowledge + current user message.
🧠
LLM REASONING ENGINE
Reads context → decides action
1
Parse user intent
2
Check available tools
3
Decide: tool call OR respond
4
Generate action / response
5
Observe result → loop or stop
while not done:
  think() → act() → observe()
⚡
ACT
Call a tool: search DB, update cart, fetch product. Real side effects happen here.
📡
OBSERVE
Tool result injected back into context. Agent evaluates if goal is achieved.
🛑
STOP CONDITIONS
Goal achieved / user responded / max iterations / human escalation needed.

System Prompt — The Agent's "Config"

## ROLE
You are a warm, caring shopping assistant for [Store Name]. Your goal is to genuinely help customers find the perfect product — not just sell.

## TONE
- Always friendly, encouraging, and empathetic
- Use light, appropriate emoji to express warmth
- Never pushy — motivate through genuine value, not pressure
- Celebrate customer decisions with positivity

## TOOLS — WHEN TO USE
- search_products(): When customer describes what they want
- get_product_detail(): When customer asks specifics about a product
- add_to_cart(): ONLY after explicit customer confirmation ("yes", "add it", "go ahead")
- checkout(): ONLY after add_to_cart succeeds AND customer confirms checkout intent
- update_inventory(): Call immediately after every successful checkout

## GUARDRAILS
- NEVER add to cart or checkout without explicit consent
- NEVER make up product details — use get_product_detail() if unsure
- If stock is 0, suggest alternatives — never promise unavailable items
- Max 3 tool call retries before escalating to human agent

## GOAL
Help customer find the right product → motivate through genuine value → complete purchase with consent → ensure DB is updated.
07 // Development Process

Agent Development Lifecycle (ADLC)

Unlike traditional SDLC where you build → test → ship, agent development is iterative, eval-driven, and never truly "done." Here are the six phases every team must go through.

PHASE 01

🎯 Problem Scoping

  • Define agent goal precisely
  • Map user journeys and intents
  • Identify required tools
  • Define success metrics (CSAT, conversion, accuracy)
  • Set scope boundaries (what agent will NOT do)
PHASE 02

🏗️ Architecture Design

  • Choose LLM model
  • Design tool schema (name, description, params)
  • Plan memory strategy (short/long-term)
  • Define context window budget
  • Design guardrails and consent flows
PHASE 03

✍️ Prompt Engineering

  • Write system prompt (role, tone, rules)
  • Create few-shot examples for edge cases
  • Define tool descriptions precisely
  • Build intent taxonomy
  • Test prompt variants manually
PHASE 04

🔧 Build & Integrate

  • Implement tool functions
  • Build agent loop / orchestrator
  • Set up RAG pipeline + vector DB
  • Integrate memory store
  • Add observability hooks
PHASE 05

🧪 Eval & Red-team

  • Build golden test dataset
  • Run LLM-as-judge evals
  • Red-team for prompt injection
  • Test edge cases & failure modes
  • Measure token cost per conversation
PHASE 06

🚀 Deploy & Monitor

  • Canary / shadow deployment
  • Set up real-time trace monitoring
  • A/B test prompt variants
  • Monitor context rot indicators
  • Continuous prompt improvement
08 // Developer Workflow

Step-by-Step Developer Workflow

A practical, ordered workflow for a software engineer building the e-commerce shopping agent from scratch to production.

WEEK 1 // SCOPING
1. Define Agent Contract
Write an "Agent Brief" — who it serves, what it can and cannot do, what tools it has access to, what the success metric is. This is your spec document.
Agent Brief Doc Intent Map Success KPIs
WEEK 1 // DESIGN
2. Design Tool Schema
Define every tool as a JSON schema: name, description (the LLM reads this to decide when to use it), required parameters, return shape. Tool descriptions are as important as the system prompt.
JSON Schema OpenAPI spec Tool descriptions
WEEK 2 // PROMPT ENGINEERING
3. Write & Iterate System Prompt
Start with role + tone + tool usage rules + guardrails. Run manual tests in the Claude Workbench / OpenAI Playground. Log what breaks. Iterate. Add few-shot examples for intent ambiguity.
Playground Testing Few-shot Examples Failure Log
WEEK 2-3 // BUILD
4. Implement Tools & Agent Loop
Build tool functions (search_products, add_to_cart, etc.) as real API/DB calls. Implement the agent loop: call LLM → parse tool call → execute → inject result → repeat. Add max iteration guard.
Python / TypeScript LangGraph / custom loop Anthropic SDK
WEEK 3 // BUILD
5. Build RAG Pipeline
Embed product catalog using an embedding model. Store in vector DB (pgvector or Pinecone). Build retrieval function: user query → semantic search → top-K products → inject into context. Test retrieval quality separately.
LlamaIndex pgvector / Pinecone Embedding Model
WEEK 3 // BUILD
6. Add Memory & Context Management
Wire Redis for short-term conversation state. Implement context compression after N turns (summarise older turns). Add context window token counter to monitor usage per call. Set alerts at 80% capacity.
Redis Summarisation Chain Token Counter
WEEK 4 // SAFETY
7. Implement Guardrails & Consent Logic
Build a consent gate: agent must receive explicit user confirmation before calling add_to_cart or checkout. Add input sanitisation for prompt injection. Add PII redaction before storing conversation. Add output validators.
Consent Gate PII Filter Input Sanitisation
WEEK 4 // OBSERVABILITY
8. Add Tracing & Monitoring
Instrument every LLM call and tool call with trace IDs. Log: token count, latency, tool called, result summary, session ID. Set up a Langfuse / LangSmith dashboard. Define alert thresholds for cost and error rate.
Langfuse OpenTelemetry Cost Alerts
WEEK 5 // EVALUATION
9. Build Eval Suite & Red-Team
Create 50+ golden test cases covering: happy path, ambiguous intent, out-of-stock scenarios, consent refusal, multi-turn correction, edge cases. Use LLM-as-judge for quality scoring. Red-team for prompt injection and jailbreaks.
Golden Dataset Red-teaming LLM-as-Judge
WEEK 6 // DEPLOY
10. Deploy with Canary Strategy
Deploy to 5% of traffic first. Monitor CSAT, conversion rate, error rate, average conversation length, cost per session. Gradually ramp. A/B test prompt variants. Set up a human escalation path for failed conversations.
Canary 5% A/B Prompts Human Fallback
ONGOING // OPERATIONS
11. Continuous Improvement Loop
Review failed conversations weekly. Add new few-shot examples for recurring failure patterns. Monitor context rot indicators. Update product embeddings as catalog changes. Refine tool descriptions as you observe misuse patterns.
Failure Review Prompt Updates Embedding Refresh

🔑 The 5 Golden Rules of Production Agent Design

01
Context is RAM — treat it like a precious resource.
Never stuff the window. Select, compress, isolate. More tokens ≠ better output.
02
Consent before every side effect.
Read operations are safe. Write operations (cart, checkout, DB update) always require explicit user confirmation.
03
Your system prompt IS your business logic.
Every rule you'd write as code belongs in the prompt. Version control it. Review it like production code.
04
Evals replace unit tests.
Build a golden dataset. Use LLM-as-judge. Ship no prompt change without running your eval suite. Probabilistic quality needs systematic measurement.
05
Always have a human fallback.
Design the escalation path on day one, not as an afterthought. Agents that fail silently destroy trust. Agents that escalate gracefully build it.
